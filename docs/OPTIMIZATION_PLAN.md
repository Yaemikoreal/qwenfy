# YuxTrans 优化规划方案

> **核心诉求**：响应速度是生命，翻译准度是底线

---

## 一、项目现状审视

### 1.1 架构优势

| 优势 | 说明 |
|------|------|
| **三层缓存架构** | Cache → Local → Cloud 自动故障转移，响应有保障 |
| **多供应商支持** | 8 个主流 AI 供应商 + 自定义 API，灵活度高 |
| **IndexedDB 持久化** | 突破 chrome.storage 5MB 限制，容量更大 |
| **LRU 内存缓存** | 热点数据 < 1ms 响应，命中率可达 100% |
| **并行翻译引擎** | 5 Worker 并发 + 视口优先，整页翻译快 |
| **文本去重优化** | 相同文本只翻译一次，节省 API 调用 |

### 1.2 当前缺陷

| 缺陷 | 影响 | 严重程度 |
|------|------|---------|
| **Service Worker 休眠** | Manifest V3 的 SW 会休眠，IndexedDB 连接断开 | 🔴 高 |
| **缓存预热缺失** | 首次翻译无缓存命中，响应慢 | 🟡 中 |
| **无离线模式** | 网络断开时完全不可用 | 🟡 中 |
| **错误提示不友好** | API 错误直接显示技术信息 | 🟡 中 |
| **无翻译质量反馈** | 用户无法标记/纠正错误翻译 | 🟡 中 |
| **设置项过多** | 新用户学习成本高 | 🟢 低 |
| **无快捷键自定义** | 固定 Ctrl+Shift+T，可能冲突 | 🟢 低 |
| **移动端不适配** | 仅支持桌面浏览器 | 🟢 低 |

### 1.3 性能现状

| 指标 | 目标 | 实测 | 差距 |
|------|------|------|------|
| 缓存命中 | < 50ms | **0.03ms** | ✅ 超预期 |
| 本地模型 | < 500ms | 未实测 | ⚠️ 依赖 Ollama |
| 云端 API | < 2s | 未实测 | ⚠️ 依赖网络 |
| 缓存命中率 | > 80% | **100%** | ✅ 超预期 |
| 整页翻译 | < 5s/页 | 未实测 | ⚠️ 待验证 |

## 二、优化提案

### 2.1 稳定性优化（P0）

#### 问题：Service Worker 休眠导致 IndexedDB 断开

**现状**：Manifest V3 的 Service Worker 在空闲 30s 后休眠，再次唤醒时 IndexedDB 连接已断开，需要重新初始化。

**优化方案**：

```
方案 A：定期保活（不推荐）
- 每 25s 发送一次心跳，阻止 SW 休眠
- 缺点：消耗资源，可能被 Chrome 限制

方案 B：懒加载重连（推荐）✅
- 每次操作前检查 db 连接状态
- 如果断开则重新打开
- 优点：零资源消耗，符合 MV3 设计理念

方案 C：chrome.storage.local 备份（推荐）✅
- IndexedDB 作为主存储
- chrome.storage.local 作为备份
- 读取时优先 IndexedDB，失败则降级到 storage.local
```

**实施建议**：
1. 在 `openDatabase()` 中添加连接状态检查
2. 实现 `ensureDBConnection()` 包装函数
3. 添加 storage.local 降级读取逻辑

### 2.2 性能优化（P1）

#### 问题：首次翻译无缓存，响应慢

**现状**：用户首次使用时缓存为空，所有翻译都需要调用 API。

**优化方案**：

```javascript
// 内置热点词库（100 个常用翻译）
const BUILTIN_CACHE = {
  'auto:zh:Hello': '你好',
  'auto:zh:World': '世界',
  'auto:zh:Thank you': '谢谢',
  'auto:zh:Welcome': '欢迎',
  'auto:zh:Sorry': '对不起',
  'auto:zh:Yes': '是',
  'auto:zh:No': '否',
  'auto:zh:OK': '好的',
  'auto:zh:Good': '好',
  'auto:zh:Bad': '坏',
  // ... 更多常用词汇
};

// 插件安装时预加载
chrome.runtime.onInstalled.addListener(() => {
  // 预加载热点词库到 IndexedDB
  preloadCache(BUILTIN_CACHE);
});
```

**预期效果**：首次使用时常用词汇命中缓存，体验更流畅。

---

#### 问题：整页翻译无进度反馈

**现状**：整页翻译时用户不知道进度，只能等待。

**优化方案**：

```javascript
// 分阶段进度提示
const PROGRESS_STAGES = {
  scanning: { text: '正在扫描页面...', progress: 10 },
  translating: { text: '正在翻译...', progress: 30 },
  rendering: { text: '正在渲染...', progress: 90 },
  done: { text: '翻译完成', progress: 100 }
};

// 显示实时进度
function updateProgress(current, total, speed) {
  const remaining = Math.ceil((total - current) / speed);
  progressEl.textContent = `已翻译 ${current}/${total} 段，预计剩余 ${remaining}s`;
}
```

#### 问题：无批量翻译优化

**现状**：整页翻译时每段文本单独请求 API。

**优化方案**：

```javascript
// 批量合并请求（OpenAI 支持）
const BATCH_SIZE = 10; // 每次最多 10 段

async function batchTranslate(texts) {
  const prompt = `Translate each of the following texts to Chinese. 
Return in JSON array format: ["translation1", "translation2", ...]

Texts:
${texts.map((t, i) => `${i + 1}. ${t}`).join('\n')}`;

  // 单次请求翻译多段
  const result = await translate(prompt);
  return JSON.parse(result);
}
```

**预期效果**：API 调用减少 80%，整页翻译速度提升 3-5 倍。

### 2.3 功能优化（P2）

#### 问题：无翻译质量反馈机制

**现状**：翻译结果无法标记或纠正。

**优化方案**：

```
用户反馈流程：
1. 翻译结果显示后，添加「👍 好」「👎 差」按钮
2. 点击「👎」后弹出纠正输入框
3. 用户输入正确翻译，提交保存到缓存
4. 统计反馈数据，用于后续优化

数据存储：
- feedback_store: { key, original, corrected, timestamp }
- 定期导出反馈数据供模型微调
```

---

#### 问题：设置项过多，新用户困惑

**现状**：设置页有 3 个标签页，选项繁多。

**优化方案**：

简化策略：

3. 高级设置折叠：默认隐藏，点击展开
4. 配置导入导出：支持分享配置文件
```

### 2.4 架构优化（P3）

#### 问题：Python 端和 JS 端代码重复

**现状**：`yuxtrans/engine/cloud.py` 和 `extension/background.js` 有大量重复逻辑。

**优化方案**：

```
方案 A：共享配置文件
- 使用 YAML/JSON 定义供应商配置
- Python 和 JS 分别读取同一配置源

方案 B：代码生成
- 从单一数据源生成 Python/JS 代码
- 保证两端逻辑一致

方案 C：后端 API（长期）
- Python 端提供统一翻译 API
- 浏览器插件通过 localhost 调用
- 优点：逻辑集中，便于维护
```

---

#### 问题：测试覆盖率不足

**现状**：浏览器插件部分无自动化测试。

**优化方案**：

测试金字塔：

    └── 工具函数、格式化测试

目标覆盖率：
- background.js: > 80%
- content.js: > 70%
- options.js: > 60%
```

| 风险 | 概率 | 影响 | 应对措施 |

## 六、总结

YuxTrans 已具备良好的架构基础，三层缓存设计保证了核心性能指标。当前最紧迫的问题是 **Service Worker 休眠导致的稳定性问题**，需要优先解决。

后续优化方向：
1. **短期**：稳定性修复 + 性能优化
2. **中期**：功能完善 + 用户体验提升
3. **长期**：架构统一 + 测试覆盖

始终保持「响应速度是生命，翻译准度是底线」的核心理念，在追求功能丰富的同时不牺牲性能。
